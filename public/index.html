1. The Big Picture: Express vs Nest

Nest uses Express as its HTTP server. When you create a Nest app, Express is running underneath — handling requests, routing, and middleware exactly as you know it. Nest adds an architectural layer on top: modules?, controllers?, services?, and dependency injection (DI)?.

Express (what you know)Nest (Express + structure)
Flat file structure, your choiceModule-based, opinionated structure
Manual wiring of routesDecorators? define routes
No built-in DIBuilt-in DI container?
app.get('/items', handler)@Get() findAll()
Middleware-centricMiddleware + Guards? + Interceptors? + Pipes?
express()NestFactory.create<NestExpressApplication>()
Key insight: Nest is not a replacement for Express — it's Express with guardrails. Your Express knowledge (middleware, req/res, routing) still applies.

2. Express Under the Hood

Nest uses @nestjs/platform-express to run Express as its HTTP layer. This is the default platform? — you don't need to install anything extra.

// main.ts — Nest explicitly using Express
import { NestFactory } from '@nestjs/core';
import { NestExpressApplication } from '@nestjs/platform-express';

const app = await NestFactory.create<NestExpressApplication>(AppModule);

Because Express is underneath, you can:

  • Use any Express middleware: app.use(cors()), app.use(helmet())
  • Access the raw Express instance: const expressApp = app.getHttpAdapter().getInstance()
  • Use @Req() and @Res() decorators? to access Express's req and res objects directly
// You can still access Express req/res when needed
@Get('custom')
customRoute(@Req() req: Request, @Res() res: Response) {
  // This is plain Express — req and res are the same objects you know
  res.status(200).json({ message: 'Hello from Express!' });
}
Bottom line: Every Nest app is an Express app. Nest just provides structure (modules, DI, decorators) so you don't have to invent your own architecture as the project grows.

3. Project Structure

A typical Nest project:

src/
├── main.ts                        // Entry point (like app.listen)
├── app.module.ts                  // Root module (wires everything)
├── items/
│   ├── items.module.ts            // Feature module
│   ├── items.controller.ts        // Route handlers
│   ├── items.controller.spec.ts   // Controller unit tests
│   ├── items.service.ts           // Business logic
│   ├── items.service.spec.ts      // Service unit tests
│   ├── item.entity.ts             // TypeORM entity
│   └── dto/
│       ├── create-item.dto.ts
│       └── update-item.dto.ts
test/
└── app.e2e-spec.ts                // End-to-end API tests

Compare with a typical Express layout:

src/
├── index.js          // app.listen + middleware
├── routes/items.js   // router.get, router.post...
├── models/item.js    // Sequelize/Mongoose model

In Nest, each feature (items, users, auth...) gets its own folder with a module, controller, and service. This keeps things organized as the app grows.

4. Modules

A Module? is a class decorated with @Module(). It groups related controllers and services together. Think of it as a self-contained feature package.

Express
// No module concept. You just
// require() and wire things manually.
const itemsRouter = require('./routes/items');
app.use('/items', itemsRouter);
Nest
@Module({
  imports: [TypeOrmModule.forFeature([Item])],
  controllers: [ItemsController],
  providers: [ItemsService],
})
export class ItemsModule {}

imports — other modules this module depends on.
controllers — route handlers.
providers? — injectable services.

The root AppModule imports all feature modules:

@Module({
  imports: [
    TypeOrmModule.forRoot({ /* db config */ }),
    ItemsModule,
  ],
})
export class AppModule {}

5. Controllers (Routes)

Controllers? handle HTTP requests. Each method maps to a route via decorators?.

Express
const router = express.Router();

router.get('/', async (req, res) => {
  const items = await Item.findAll();
  res.json(items);
});

router.post('/', async (req, res) => {
  const item = await Item.create(req.body);
  res.status(201).json(item);
});

router.get('/:id', async (req, res) => {
  const item = await Item.findByPk(req.params.id);
  res.json(item);
});
Nest
@Controller('items')
export class ItemsController {
  constructor(
    private readonly itemsService: ItemsService
  ) {}

  @Get()
  findAll() {
    return this.itemsService.findAll();
  }

  @Post()
  create(@Body() dto: CreateItemDto) {
    return this.itemsService.create(dto);
  }

  @Get(':id')
  findOne(@Param('id') id: number) {
    return this.itemsService.findOne(id);
  }
}
Notice: In Nest, controllers don't touch the database directly. They delegate to a service?. This separation makes testing much easier.

6. Services & Dependency Injection

A Service? holds your business logic. It's decorated with @Injectable()? and gets automatically injected into controllers via the constructor.

Express
// Typically mixed into route handler,
// or a plain module you require()
const db = require('../db');

async function getAllItems() {
  return db.query('SELECT * FROM items');
}

module.exports = { getAllItems };
Nest
@Injectable()
export class ItemsService {
  constructor(
    @InjectRepository(Item)
    private readonly repo: Repository<Item>,
  ) {}

  findAll(): Promise<Item[]> {
    return this.repo.find();
  }

  async findOne(id: number): Promise<Item> {
    const item = await this.repo.findOneBy({ id });
    if (!item) throw new NotFoundException();
    return item;
  }
}

How DI works: You declare the service in a module's providers? array. Nest's DI container? creates a singleton? and injects it wherever it's requested via constructor parameters. No manual require() or new Service() needed.

8. TypeORM Integration

TypeORM handles database operations. An Entity? maps a class to a database table.

import { Entity, PrimaryGeneratedColumn, Column } from 'typeorm';

@Entity()
export class Item {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  name: string;

  @Column({ default: '' })
  description: string;
}

Wiring it up:

  • TypeOrmModule.forRoot({...})? in AppModule — database connection config
  • TypeOrmModule.forFeature([Item])? in ItemsModule — registers the entity for that module
  • @InjectRepository(Item)? in the service — injects the TypeORM repository?

The repository? gives you methods like .find(), .findOneBy(), .save(), .remove() — no raw SQL needed.

9. Validation with Pipes

Nest uses Pipes? to transform and validate incoming data. The ValidationPipe? works with class-validator decorators on DTOs?.

Express (manual)
router.post('/', (req, res) => {
  if (!req.body.name) {
    return res.status(400)
      .json({ error: 'Name required' });
  }
  // proceed...
});
Nest (declarative)
export class CreateItemDto {
  @IsString()
  @IsNotEmpty()
  name: string;

  @IsString()
  @IsOptional()
  description?: string;
}

Enable it globally in main.ts:

app.useGlobalPipes(new ValidationPipe({ whitelist: true }));

whitelist: true? strips any properties not in the DTO?. Validation errors automatically return a 400 response.

10. Testing

Nest has first-class testing support via @nestjs/testing. The key idea: you create a testing module? where you can swap real dependencies with mocks.

Unit Test (Service)

const module = await Test.createTestingModule({
  providers: [
    ItemsService,
    {
      provide: getRepositoryToken(Item),
      useValue: mockRepository,  // fake repo
    },
  ],
}).compile();

const service = module.get(ItemsService);
const result = await service.findAll();
expect(result).toEqual([mockItem]);

Unit Test (Controller)

const module = await Test.createTestingModule({
  controllers: [ItemsController],
  providers: [
    { provide: ItemsService, useValue: mockService },
  ],
}).compile();

const controller = module.get(ItemsController);
expect(await controller.findAll()).toEqual([mockItem]);

E2E Test

const module = await Test.createTestingModule({
  imports: [AppModule],
}).compile();

const app = module.createNestApplication();
await app.init();

await request(app.getHttpServer())
  .post('/items')
  .send({ name: 'Widget' })
  .expect(201);
Key difference from Express testing: In Express, you'd use supertest(app) directly. In Nest, you build a testing module? first, then call app.getHttpServer() to get the underlying Express server for supertest. This extra step lets you easily swap out databases, services, or any dependency.

11. Running This Project

This page is served by the very NestJS app it describes. Here's how to set it up:

# 1. Create the database
createdb nest_crud
createdb nest_crud_test  # for e2e tests

# 2. Install & run
npm install
npm run start:dev

# 3. Test the API
curl -X POST http://localhost:3000/items \
  -H "Content-Type: application/json" \
  -d '{"name":"Widget","description":"A cool widget"}'

curl http://localhost:3000/items

# 4. Run tests
npm test              # unit tests
npm run test:e2e      # e2e tests (needs nest_crud_test db)
