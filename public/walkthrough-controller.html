Step 6: The Controller
items.controller.ts

The controller is the HTTP layer. It receives requests, extracts data from them (body, params, query), delegates to the service, and returns the result. Nest handles serializing the return value to JSON and setting the correct status code.

src/items/items.controller.ts
import {
  Controller, Get, Post, Put, Delete,
  Body, Param, ParseIntPipe,
} from '@nestjs/common';
import { ItemsService } from './items.service.js';
import { CreateItemDto } from './dto/create-item.dto.js';
import { UpdateItemDto } from './dto/update-item.dto.js';

@Controller('items')
export class ItemsController {
  constructor(private readonly itemsService: ItemsService) {}

  @Post()
  create(@Body() createItemDto: CreateItemDto) {
    return this.itemsService.create(createItemDto);
  }

  @Get()
  findAll() {
    return this.itemsService.findAll();
  }

  @Get(':id')
  findOne(@Param('id', ParseIntPipe) id: number) {
    return this.itemsService.findOne(id);
  }

  @Put(':id')
  update(
    @Param('id', ParseIntPipe) id: number,
    @Body() updateItemDto: UpdateItemDto,
  ) {
    return this.itemsService.update(id, updateItemDto);
  }

  @Delete(':id')
  remove(@Param('id', ParseIntPipe) id: number) {
    return this.itemsService.remove(id);
  }
}

The Class Setup

1

@Controller('items') — the route prefix

@Controller('items')
export class ItemsController {

The string 'items' sets the route prefix for every method in this controller. All routes will start with /items.

Express equivalent: 
const router = express.Router();
app.use('/items', router);
Same idea — the prefix is declared once and applies to all routes in the group.
2

Constructor injection

constructor(private readonly itemsService: ItemsService) {}

Nest sees that this controller needs an ItemsService and automatically provides it. You never create the service yourself. This is the same DI pattern from the service, but here the controller is the consumer.

Because the service is injected (not imported directly), you can swap it with a mock in tests — which is exactly what we do in the controller tests.

The Route Methods

3

POST /items — Create

@Post()
create(@Body() createItemDto: CreateItemDto) {
  return this.itemsService.create(createItemDto);
}

Let's trace what happens when a client sends POST /items { "name": "Widget" }:

  1. Express receives the request (it's still Express under the hood!)
  2. @Post() matches the HTTP method and route (/items from the controller prefix + no sub-path)
  3. @Body() extracts req.body from the Express request
  4. The ValidationPipe validates the body against CreateItemDto
  5. The validated DTO is passed to create()
  6. The service creates the item and returns a Promise
  7. Nest awaits the Promise and sends the result as JSON with status 201 (POST default)
Nest automatically returns 201 for POST and 200 for GET/PUT/DELETE. You can override with @HttpCode(204) on the method.
4

GET /items — List all

@Get()
findAll() {
  return this.itemsService.findAll();
}

The simplest route. No parameters, no body. Just call the service and return the array. Nest serializes the array to JSON automatically.

Express equivalent: 
router.get('/', async (req, res) => {
  const items = await itemsService.findAll();
  res.json(items);
});
In Nest, you don't call res.json() — just return the value and Nest handles the response.
5

GET /items/:id — Get one (with ParseIntPipe)

@Get(':id')
findOne(@Param('id', ParseIntPipe) id: number) {
  return this.itemsService.findOne(id);
}

@Param('id') extracts req.params.id from the URL. But URL params are always strings in Express. ParseIntPipe converts it to a number automatically.

If someone requests GET /items/abc, the ParseIntPipe returns a 400 error:

{
  "statusCode": 400,
  "message": "Validation failed (numeric string is expected)",
  "error": "Bad Request"
}
Express equivalent: 
router.get('/:id', async (req, res) => {
  const id = parseInt(req.params.id, 10);
  if (isNaN(id)) {
    return res.status(400).json({ error: 'Invalid ID' });
  }
  const item = await itemsService.findOne(id);
  res.json(item);
});
ParseIntPipe replaces 3 lines of manual parsing + validation with a single pipe argument.
6

PUT /items/:id — Update

@Put(':id')
update(
  @Param('id', ParseIntPipe) id: number,
  @Body() updateItemDto: UpdateItemDto,
) {
  return this.itemsService.update(id, updateItemDto);
}

This method combines both patterns: extracting a route param AND a request body. The :id param is parsed to a number, the body is validated against UpdateItemDto, and both are passed to the service.

7

DELETE /items/:id — Remove

@Delete(':id')
remove(@Param('id', ParseIntPipe) id: number) {
  return this.itemsService.remove(id);
}

Same pattern as GET one — parse the ID, delegate to the service. Since remove() returns void, the response body will be empty with a 200 status.

The Full Route Table

Here's what Nest registers on the Express router from this controller:

What the Controller Does NOT Do

  • No database access. That's the service's job.
  • No validation logic. That's handled by DTOs + ValidationPipe.
  • No error response formatting. Nest exceptions (NotFoundException, etc.) handle that.
  • No res.json() or res.status(). Just return the value and Nest handles it.

The controller is deliberately thin. It's a mapping layer: HTTP in, service call, data out.

  │

  │

ItemsService.create(dto) → INSERT into PostgreSQL
  │

Nest awaits the Promise, serializes to JSON
  │
  ▼
Client receives: 201 { "id": 1, "name": "Widget", "description": "" }
Key takeaway: The controller is a thin routing layer. Decorators replace Express's manual router.get() / req.params / res.json() pattern. Every method follows the same shape: extract inputs via decorators, call the service, return the result.
