Step 5: The Service
items.service.ts

The service is where your business logic lives. It talks to the database via the TypeORM repository and exposes methods that the controller calls. It has no knowledge of HTTP — no req, no res, no status codes. Just pure data operations.

src/items/items.service.ts
import { Injectable, NotFoundException } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { Item } from './item.entity.js';
import { CreateItemDto } from './dto/create-item.dto.js';
import { UpdateItemDto } from './dto/update-item.dto.js';

@Injectable()
export class ItemsService {
  constructor(
    @InjectRepository(Item)
    private readonly itemsRepository: Repository<Item>,
  ) {}

  create(createItemDto: CreateItemDto): Promise<Item> {
    const item = this.itemsRepository.create(createItemDto);
    return this.itemsRepository.save(item);
  }

  findAll(): Promise<Item[]> {
    return this.itemsRepository.find();
  }

  async findOne(id: number): Promise<Item> {
    const item = await this.itemsRepository.findOneBy({ id });
    if (!item) {
      throw new NotFoundException(`Item #${id} not found`);
    }
    return item;
  }

  async update(id: number, updateItemDto: UpdateItemDto): Promise<Item> {
    const item = await this.findOne(id);
    Object.assign(item, updateItemDto);
    return this.itemsRepository.save(item);
  }

  async remove(id: number): Promise<void> {
    const item = await this.findOne(id);
    await this.itemsRepository.remove(item);
  }
}

The Class Setup

1

@Injectable()

@Injectable()
export class ItemsService {

This decorator tells Nest's DI container: "this class can be injected into other classes." Without it, trying to inject ItemsService into a controller would throw an error at startup. Every service, every repository, every custom provider needs this decorator.

2

Constructor injection

constructor(
  @InjectRepository(Item)
  private readonly itemsRepository: Repository<Item>,
) {}

This is where DI happens. Let's unpack it:

  • @InjectRepository(Item) — tells Nest to inject the TypeORM repository for the Item entity
  • private readonly — TypeScript shorthand that creates a class property and assigns it in one line
  • Repository<Item> — a TypeORM generic class that provides all the database methods for the Item table

You never call new ItemsService() yourself. Nest sees that ItemsController needs an ItemsService, which needs a Repository<Item>, and automatically creates and connects them.

Express equivalent: 
// In Express, you'd manually wire this:
const pool = require('../db');
// Then use pool.query() in every function
Nest's DI removes this manual wiring. The repository comes pre-connected.

The CRUD Methods

3

create() — INSERT

create(createItemDto: CreateItemDto): Promise<Item> {
  const item = this.itemsRepository.create(createItemDto);
  return this.itemsRepository.save(item);
}

Two steps here:

  1. repository.create(dto) — creates an Item instance in memory from the DTO. Does NOT touch the database yet. It's like new Item() with the DTO values assigned.
  2. repository.save(item) — actually inserts the row into PostgreSQL and returns the saved entity with the auto-generated id.

The generated SQL is:

INSERT INTO item (name, description)
VALUES ('Widget', 'A cool widget')
RETURNING id, name, description;
4

findAll() — SELECT *

findAll(): Promise<Item[]> {
  return this.itemsRepository.find();
}

The simplest method. repository.find() with no arguments returns all rows. The generated SQL:

SELECT * FROM item;

You can pass options to filter: .find({ where: { name: 'Widget' } }), .find({ order: { id: 'DESC' }, take: 10 }), etc.

5

findOne() — SELECT by ID with error handling

async findOne(id: number): Promise<Item> {
  const item = await this.itemsRepository.findOneBy({ id });
  if (!item) {
    throw new NotFoundException(`Item #${id} not found`);
  }
  return item;
}

findOneBy({ id }) returns the matching row or null. If null, we throw a NotFoundException — a built-in Nest exception that automatically returns a 404 response:

// Automatic 404 response
{
  "statusCode": 404,
  "message": "Item #999 not found",
  "error": "Not Found"
}
Express equivalent: 
if (!item) {
  return res.status(404).json({ error: 'Not found' });
}
In Nest, you throw an exception and the framework handles the HTTP response. No res object needed.

This method is also reused by update() and remove() to verify the item exists before modifying it.

6

update() — UPDATE

async update(id: number, updateItemDto: UpdateItemDto): Promise<Item> {
  const item = await this.findOne(id);
  Object.assign(item, updateItemDto);
  return this.itemsRepository.save(item);
}

Three steps:

  1. this.findOne(id) — fetch the existing item (throws 404 if not found)
  2. Object.assign(item, updateItemDto) — merge the DTO's properties into the entity. Since UpdateItemDto fields are optional, only provided fields get overwritten
  3. repository.save(item) — because the entity already has an id, TypeORM performs an UPDATE instead of an INSERT

If the client sends { "name": "Updated" }, only the name changes. The description stays the same.

7

remove() — DELETE

async remove(id: number): Promise<void> {
  const item = await this.findOne(id);
  await this.itemsRepository.remove(item);
}

Fetches the item first (404 if missing), then deletes it. The return type is Promise<void> — nothing to return after deletion. The generated SQL:

DELETE FROM item WHERE id = 1;

Key Design Decisions

  • No HTTP concepts. The service doesn't know about requests, responses, or status codes. It throws NotFoundException (which Nest maps to 404), but it doesn't construct HTTP responses.
  • Reuses findOne(). Both update() and remove() call findOne() to verify the item exists. DRY, and consistent 404 handling.
  • Typed with DTOs. The create() method takes CreateItemDto, not any. TypeScript ensures you can only pass validated data.
  • Returns Promises. All methods are async, matching the reality that database operations are I/O-bound. Nest automatically awaits the returned Promise and sends the resolved value as the response body.
Key takeaway: The service is a clean data layer with no HTTP knowledge. It receives typed DTOs, talks to the database via the repository, and throws Nest exceptions for error cases. The controller handles the HTTP part; the service handles the data part.
