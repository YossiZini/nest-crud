Step 7: Testing
Unit Tests & E2E Tests

Nest's DI system makes testing straightforward: create a testing module, swap real dependencies with mocks, and test each layer in isolation. Our project has three test files — let's walk through each one in detail.

Service Unit Tests

The service talks to the database via the repository. In tests, we replace the real repository with a mock object so we can test the service logic without PostgreSQL.

src/items/items.service.spec.ts
import { Test, TestingModule } from '@nestjs/testing';
import { getRepositoryToken } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { NotFoundException } from '@nestjs/common';
import { ItemsService } from './items.service';
import { Item } from './item.entity';

Setting Up the Testing Module

1

The mock data and mock repository

const mockItem: Item = { id: 1, name: 'Test', description: 'Desc' };

const mockRepository = {
  create:    jest.fn().mockReturnValue(mockItem),
  save:      jest.fn().mockResolvedValue(mockItem),
  find:      jest.fn().mockResolvedValue([mockItem]),
  findOneBy: jest.fn().mockResolvedValue(mockItem),
  remove:    jest.fn().mockResolvedValue(mockItem),
};

We create a fake repository object with the same methods the service calls. Each method is a Jest mock that returns predictable data. Notice:

  • mockReturnValue — for synchronous methods (create() returns an entity, not a Promise)
  • mockResolvedValue — for async methods (returns a resolved Promise)
Why mock the repository? We're testing the service logic, not the database. We want to verify that the service calls the right repository methods with the right arguments, handles null results correctly, etc. — without needing a real PostgreSQL instance.
2

Building the testing module

beforeEach(async () => {
  const module: TestingModule = await Test.createTestingModule({
    providers: [
      ItemsService,
      { provide: getRepositoryToken(Item), useValue: mockRepository },
    ],
  }).compile();

  service = module.get<ItemsService>(ItemsService);
  repo = module.get<Repository<Item>>(getRepositoryToken(Item));
});

This is the core Nest testing pattern. Let's break it down:

  1. Test.createTestingModule() — creates a module just like @Module(), but for testing
  2. providers: [ItemsService, ...] — we include the real ItemsService (that's what we're testing)
  3. { provide: getRepositoryToken(Item), useValue: mockRepository } — this is the swap. Instead of the real TypeORM repository, inject our mock. getRepositoryToken(Item) returns the same DI token that @InjectRepository(Item) uses
  4. .compile() — builds the module and creates all instances
  5. module.get() — retrieves instances from the DI container for use in tests
Express equivalent: In Express, you'd typically have to restructure your code to accept dependencies as function parameters, or use a library like proxyquire/jest.mock() to stub require() calls. Nest's DI makes this swap trivial — no code changes needed.
3

Cleanup between tests

afterEach(() => jest.clearAllMocks());

Resets all mock call counts and arguments between tests so they don't bleed into each other.

The Test Cases

4

Testing create()

it('should create an item', async () => {
  const dto = { name: 'Test', description: 'Desc' };
  const result = await service.create(dto);
  expect(repo.create).toHaveBeenCalledWith(dto);
  expect(repo.save).toHaveBeenCalled();
  expect(result).toEqual(mockItem);
});

We verify three things:

  1. The service called repository.create() with the DTO
  2. The service called repository.save() to persist it
  3. The returned value matches what the repository returned
5

Testing the 404 case

it('should throw NotFoundException', async () => {
  mockRepository.findOneBy.mockResolvedValueOnce(null);
  await expect(service.findOne(999)).rejects.toThrow(NotFoundException);
});

Here we override the mock for one call only (mockResolvedValueOnce) to return null, simulating a missing item. Then we verify the service throws NotFoundException. This tests our error handling without touching the database.

6

Testing update()

it('should update an item', async () => {
  const updated = { ...mockItem, name: 'Updated' };
  mockRepository.save.mockResolvedValueOnce(updated);
  const result = await service.update(1, { name: 'Updated' });
  expect(result.name).toBe('Updated');
});

We mock save() to return an updated item and verify the result reflects the change.

Controller Unit Tests

The controller tests swap the real service with a mock, so we're only testing the controller's routing and delegation logic.

src/items/items.controller.spec.ts
const mockItem = { id: 1, name: 'Test', description: 'Desc' };

const mockService = {
  create:  jest.fn().mockResolvedValue(mockItem),
  findAll: jest.fn().mockResolvedValue([mockItem]),
  findOne: jest.fn().mockResolvedValue(mockItem),
  update:  jest.fn().mockResolvedValue({ ...mockItem, name: 'Updated' }),
  remove:  jest.fn().mockResolvedValue(undefined),
};
7

Swapping the service

const module: TestingModule = await Test.createTestingModule({
  controllers: [ItemsController],
  providers: [{ provide: ItemsService, useValue: mockService }],
}).compile();

Same pattern as the service test, but this time:

  • We include the real ItemsController (what we're testing)
  • We swap ItemsService with mockService

The controller has constructor(private readonly itemsService: ItemsService). Nest injects our mock instead of the real service.

8

Testing delegation

it('should create an item', async () => {
  const dto = { name: 'Test', description: 'Desc' };
  expect(await controller.create(dto)).toEqual(mockItem);
  expect(service.create).toHaveBeenCalledWith(dto);
});

We verify two things:

  1. The controller returns what the service returns (it's a pass-through)
  2. The controller called the correct service method with the correct argument

The controller tests are simpler than the service tests because the controller is a thin layer — it just delegates.

E2E Tests

End-to-end tests test the full stack: HTTP request → controller → service → database. They use a real PostgreSQL database (a test database) and make real HTTP calls via supertest.

test/app.e2e-spec.ts
import { Test, TestingModule } from '@nestjs/testing';
import { INestApplication, ValidationPipe } from '@nestjs/common';
import request from 'supertest';
import { TypeOrmModule } from '@nestjs/typeorm';
import { ItemsModule } from '../src/items/items.module.js';
import { Item } from '../src/items/item.entity.js';
9

Setting up the test app

beforeAll(async () => {
  const moduleFixture: TestingModule = await Test.createTestingModule({
    imports: [
      TypeOrmModule.forRoot({
        type: 'postgres',
        host: process.env.DB_HOST || 'localhost',
        port: parseInt(process.env.DB_PORT, 10) || 5432,
        username: process.env.DB_USER || 'postgres',
        password: process.env.DB_PASS || 'postgres',
        database: process.env.DB_NAME || 'nest_crud_test',
        entities: [Item],
        synchronize: true,
        dropSchema: true,
      }),
      ItemsModule,
    ],
  }).compile();

  app = moduleFixture.createNestApplication();
  app.useGlobalPipes(new ValidationPipe({ whitelist: true }));
  await app.init();
});

Key differences from unit tests:

  • Real database. We connect to nest_crud_test (a separate test database)
  • dropSchema: true — drops all tables before tests run, ensuring a clean slate
  • createNestApplication() — creates a full Nest app (with the Express server) that we can send HTTP requests to
  • useGlobalPipes() — we need to add the validation pipe again because e2e tests create a fresh app instance
  • beforeAll instead of beforeEach — we set up once because the tests run sequentially and build on each other (create, then read, then update, then delete)
10

Testing the full CRUD flow

let createdId: number;

it('POST /items', async () => {
  const res = await request(app.getHttpServer())
    .post('/items')
    .send({ name: 'Widget', description: 'A test widget' })
    .expect(201);

  expect(res.body.name).toBe('Widget');
  createdId = res.body.id;
});

app.getHttpServer() returns the underlying Express server instance. We pass it to supertest's request() to make real HTTP calls. This is the exact same supertest you'd use in Express tests.

We save the createdId to use in subsequent tests (read, update, delete).

11

Testing the full lifecycle

The remaining tests follow the CRUD flow in order:

it('GET /items', ...)        // list all → expect 1 item
it('GET /items/:id', ...)    // get by id → expect 'Widget'
it('PUT /items/:id', ...)    // update name → expect 'Updated Widget'
it('DELETE /items/:id', ...)  // delete → expect 200
it('GET /items/:id → 404')   // verify it's gone → expect 404

This tests the entire stack: HTTP → validation → controller → service → repository → PostgreSQL and back. The final test (404 after delete) verifies the delete actually worked.

12

Cleanup

afterAll(async () => {
  await app.close();
});

Closes the Nest app, which disconnects from the database and shuts down the Express server. Without this, the test process would hang.

Running the Tests

# Unit tests (no database needed)
npm test

# E2E tests (needs nest_crud_test database)
createdb nest_crud_test    # run once
npm run test:e2e

Key takeaway: Nest's testing story is built on DI. You create a testing module, swap real dependencies with mocks at the provider level, and test each layer independently. The same DI container that powers your app powers your tests — no hacks, no monkey-patching, no jest.mock() gymnastics.
