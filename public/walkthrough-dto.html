Step 4: DTOs
Data Transfer Objects

A DTO defines the shape of data coming into your API. While the entity defines what's in the database, DTOs define what clients are allowed to send. They're also where validation rules live.

We have two DTOs — one for creating items, one for updating them. Each operation has different rules: creating requires a name, updating doesn't.

CreateItemDto

src/items/dto/create-item.dto.ts
import { IsString, IsNotEmpty, IsOptional } from 'class-validator';

export class CreateItemDto {
  @IsString()
  @IsNotEmpty()
  name: string;

  @IsString()
  @IsOptional()
  description?: string;
}
1

name — required string

@IsString()
@IsNotEmpty()
name: string;

Two validation decorators stacked on one property:

  • @IsString() — rejects anything that isn't a string (numbers, booleans, objects, etc.)
  • @IsNotEmpty() — rejects empty strings "" and missing values

If a client sends { "name": "" } or { "name": 123 } or omits name entirely, the ValidationPipe returns a 400 Bad Request with a detailed error message before your controller code ever runs.

Express equivalent: 
router.post('/', (req, res) => {
  if (!req.body.name || typeof req.body.name !== 'string') {
    return res.status(400).json({
      error: 'name must be a non-empty string'
    });
  }
  // proceed...
});
In Nest, those 5 lines of manual validation become 2 decorators.
2

description — optional string

@IsString()
@IsOptional()
description?: string;
  • @IsOptional() — the field can be omitted entirely (no error)
  • @IsString() — but if provided, it must be a string

The ? in description?: string is TypeScript's optional property syntax. Combined with @IsOptional(), this means the client can send either { "name": "Widget" } or { "name": "Widget", "description": "A cool widget" }.

UpdateItemDto

src/items/dto/update-item.dto.ts
import { IsString, IsOptional } from 'class-validator';

export class UpdateItemDto {
  @IsString()
  @IsOptional()
  name?: string;

  @IsString()
  @IsOptional()
  description?: string;
}

Notice the difference from CreateItemDto: both fields are optional. When updating, the client should be able to send just the fields they want to change:

// Update only the name
PUT /items/1
{ "name": "Updated Widget" }

// Update only the description
PUT /items/1
{ "description": "New description" }

// Update both
PUT /items/1
{ "name": "Updated", "description": "New desc" }

Why Separate DTOs?

You might wonder: "Why not reuse the entity class for validation?" Three reasons:

  1. Different rules per operation. Creating requires name, updating doesn't. The entity doesn't know about these rules.
  2. Security. The entity has an id field. If you used the entity as the DTO, a client could send { "id": 999 } and potentially overwrite another record. DTOs expose only the fields clients should control.
  3. Whitelist protection. Because we set whitelist: true in the ValidationPipe, any property NOT in the DTO gets stripped. If someone sends { "name": "Widget", "role": "admin" }, the role field is silently removed.

What Happens When Validation Fails

If a client sends invalid data, the ValidationPipe returns a 400 response with structured error details:

// Request
POST /items
{ "name": 123 }

// Response — 400 Bad Request
{
  "statusCode": 400,
  "message": [
    "name must be a string"
  ],
  "error": "Bad Request"
}

Your controller code never runs. The validation happens in the pipe layer, before the request reaches the controller.

The Flow: Request → Pipe → Controller

Client sends POST /items { "name": "Widget" }
       │
       ▼
ValidationPipe checks body against CreateItemDto
       │
       ├─ Valid?   → passes to ItemsController.create()
       │
       └─ Invalid? → returns 400 immediately
Key takeaway: DTOs are the gatekeepers of your API. They define exactly what data is allowed in and validate it automatically. One DTO per operation gives you fine-grained control over what each endpoint accepts.
