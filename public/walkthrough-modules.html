Step 2: Modules
The Wiring System

Modules are the organizing principle of Nest. Every Nest app has at least one module (the root AppModule). Each feature gets its own module. Think of a module as a box that holds everything related to one feature: its routes, its business logic, and its database access.

The Root Module: AppModule

src/app.module.ts
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { ServeStaticModule } from '@nestjs/serve-static';
import { join } from 'path';
import { ItemsModule } from './items/items.module.js';
import { Item } from './items/item.entity.js';

@Module({
  imports: [
    ServeStaticModule.forRoot({
      rootPath: join(__dirname, '..', 'public'),
    }),
    TypeOrmModule.forRoot({
      type: 'postgres',
      host: process.env.DB_HOST || 'localhost',
      port: parseInt(process.env.DB_PORT ?? '5432', 10),
      username: process.env.DB_USER || 'postgres',
      password: process.env.DB_PASS || 'postgres',
      database: process.env.DB_NAME || 'nest_crud',
      entities: [Item],
      synchronize: true,
    }),
    ItemsModule,
  ],
})
export class AppModule {}

The AppModule is the root of the module tree. It doesn't have its own controllers or services — its job is to wire together global configuration and feature modules.

Breaking it down

1

The @Module() decorator

@Module({
  imports: [ ... ],
})

The @Module() decorator takes a configuration object. The most important property is imports — an array of other modules this module depends on. Since AppModule is the root, it imports everything the app needs.

Express equivalent: In Express, your index.js would have a series of app.use() calls for each router, plus database setup code, plus static file config. The @Module decorator replaces all of that wiring.
2

ServeStaticModule

ServeStaticModule.forRoot({
  rootPath: join(__dirname, '..', 'public'),
}),

This serves static files from the public/ directory. It's the Nest equivalent of app.use(express.static('public')). The .forRoot() pattern means "configure this module at the root level with these settings."

3

TypeOrmModule.forRoot()

TypeOrmModule.forRoot({
  type: 'postgres',
  host: process.env.DB_HOST || 'localhost',
  // ... connection config
  entities: [Item],
  synchronize: true,
}),

This establishes the PostgreSQL connection for the entire app. Key properties:

  • type: 'postgres' — which database driver to use
  • entities: [Item] — tells TypeORM which entity classes to manage
  • synchronize: true — auto-creates/updates tables to match your entities. Only use this in development — in production, use migrations
Express equivalent: You'd typically call createConnection() or new Pool() somewhere in your startup file and pass the connection around manually. Here, Nest's DI container manages the connection and makes it available to any module that needs it.
4

Importing the feature module

ItemsModule,

This single line tells Nest: "include the Items feature in this app." All of the Items routes, services, and database access come along automatically. No need to manually register each route.

5

The empty class body

export class AppModule {}

The class itself is empty! All the configuration lives in the @Module() decorator. Nest reads the decorator metadata at startup to wire everything together. The class is just a container for that metadata.

The Feature Module: ItemsModule

src/items/items.module.ts
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { Item } from './item.entity.js';
import { ItemsController } from './items.controller.js';
import { ItemsService } from './items.service.js';

@Module({
  imports: [TypeOrmModule.forFeature([Item])],
  controllers: [ItemsController],
  providers: [ItemsService],
})
export class ItemsModule {}

This is a feature module — it encapsulates everything related to the "items" feature. Let's break down the three properties:

1

imports: [TypeOrmModule.forFeature([Item])]

While forRoot() in AppModule sets up the database connection, forFeature() registers specific entities for this module. This makes Repository<Item> injectable within ItemsModule. Without this line, the service wouldn't be able to access the Item table.

Think of it this way: forRoot() = "connect to the database." forFeature() = "this module needs access to these tables."
2

controllers: [ItemsController]

Declares which controllers belong to this module. Nest will scan ItemsController for route decorators (@Get(), @Post(), etc.) and register them on the Express router. All routes will be prefixed with whatever path is set in @Controller('items').

3

providers: [ItemsService]

Declares which services are available for injection within this module. When ItemsController's constructor asks for an ItemsService, the DI container knows to provide the instance declared here.

Express equivalent: There's no direct equivalent. In Express, you'd just require('./items.service') in your route file. Nest's provider system adds testability — you can easily swap ItemsService with a mock in tests.

How the Module Tree Works

Here's the full picture of how modules connect in our app:

AppModule (root)
├── ServeStaticModule      → serves public/ files
├── TypeOrmModule.forRoot   → PostgreSQL connection
└── ItemsModule            (feature)
    ├── TypeOrmModule.forFeature  → Item repository
    ├── ItemsController          → /items routes
    └── ItemsService             → business logic

When you add a new feature (e.g., users), you create a new UsersModule with its own controller and service, then add it to AppModule's imports. The feature is completely self-contained — you can even remove it by deleting one line from the imports array.

Key takeaway: Modules are the backbone of a Nest app. They enforce separation of concerns and make features portable. Each module is a self-contained unit that declares exactly what it needs and what it provides.
